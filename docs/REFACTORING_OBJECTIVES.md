# 리팩토링 목적 리포트

## 배경
- 로컬 Supabase 스택과 Python GUI 업로더로 압출 공정 데이터를 적재·검증.
- 데이터/로그/도구는 분리했지만, 핵심 로직이 여전히 혼재하고 일회성 스크립트 의존도가 높음.

## 목표
1) 신뢰성: 업로드/재시도/재개(resume) 흐름과 헤더(apikey) 요구사항을 표준화해 중복·누락 방지.
2) 가독성·유지보수: GUI와 핵심 로직을 계층화하고 타입 힌트·에러 메시지를 정비.
3) 배포 용이성: pyinstaller 빌드 설정을 표준화하고 환경 주입(.env, config.ini)을 명시.
4) 관측성: 로깅·검증 스크립트를 통합해 ETL/뷰 성능 문제를 조기 감지.
5) 테스트 가능성: 샘플 데이터 기반 단위/통합 스모크 테스트 진입점을 마련해 회귀를 빠르게 확인.

## 범위
- 포함: core 업로드/변환 모듈, GUI 엔트리, Supabase REST/Edge 연동, 로깅·검증 스크립트, 빌드 스펙.
- 제외: 스키마 변경, 백업/운영 정책 조정, Grafana 대시보드 콘텐츠.

## 우선순위(상→하)
1) 업로드 경로 강화: 재시도·resume·헤더 일관성, 대용량 CSV를 위한 메모리 안전 배치.
2) 구조 정리: GUI 업로드 호출 경로와 설정 로딩·검증 단일화.
3) 로깅/관측: 구조화 로그와 일관된 성공/오류 포맷으로 통일.
4) 테스트 진입점: `scripts/verification/*`을 샘플 데이터 기반 스모크 테스트로 표준화.
5) 빌드/배포: pyinstaller 스펙 단순화, dist 출력과 설정 템플릿 동기화.

## 성공 기준
- 동일 입력 시 결과가 동등 또는 개선(중복 감소, 실패율 감소).
- GUI가 공통 모듈을 재사용해 코드 중복 20%+ 감소.
- 검증 스크립트가 CI 또는 수동 체크리스트로 실행 가능하고, 명확한 패스/페일을 제공.
- 빌드가 1~2개 명령으로 완료되며, 환경 누락 시 명확한 오류 메시지.
- 로그/오류가 `logs/`에 모이고 심각도 태깅이 구분됨.

## 리스크 및 완화
- 스키마 의존: 코드 리팩토링에 집중하고, 마이그레이션은 별도 커밋으로 분리.
- 대용량 처리: 배치/타임아웃 조정 시 샘플 데이터 회귀 테스트 필수.
- 빌드 환경차: Windows/WSL에서 DLL·아이콘 경로 확인, 소규모 빌드 매트릭스 점검.
